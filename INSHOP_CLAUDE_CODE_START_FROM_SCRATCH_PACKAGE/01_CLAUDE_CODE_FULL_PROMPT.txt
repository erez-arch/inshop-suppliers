CORE FLOW TO IMPLEMENT:

1. Supplier side:
Supplier opens mobile flow.
Supplier uploads/scans debit invoice.
Supplier uploads/takes photos of goods.
Supplier chooses branch.
System parses invoice through AIInvoiceParserAdapter.
System validates supplier, branch, invoice number, date, total and line items.
System sends mock WhatsApp/SMS notification to the trustee.
Delivery status becomes reported_supplier.

2. Trustee side:
Trustee opens personal receiving link.
System knows the trustee from link/login.
Trustee sees supplier delivery.
Trustee scans/approves invoice.
Trustee uploads/takes photos of received goods.
Trustee sees invoice lines parsed by AI.
Trustee confirms each line quantity or changes quantity actually received.
Trustee can report missing quantities.
Trustee uploads empty pantry photo and full pantry photo.
Trustee completes receiving.
System calculates trustee reward = 2% of invoice value.
Receiving status becomes trustee_completed.
This does not yet update final inventory.

3. Admin receiving side:
Admin sees all pending receiving records.
Admin sees:
- supplier invoice image
- trustee invoice image
- supplier goods photos
- trustee goods photos
- empty pantry photo
- full pantry photo
- AI confidence
- branch/supplier mismatch warnings
- all invoice line items
- invoice quantity
- trustee received quantity
- admin final inventory quantity
- shortage amount
- credit request amount

Admin can:
- add item
- remove item
- search INSHOP item
- correct item mapping
- correct quantity entering inventory
- approve inventory update
- create credit request
- send credit request to supplier
- approve override when needed

Final inventory may be updated only after admin approval.
Status becomes admin_approved or credit_requested when relevant.

4. Unknown item matching:
If invoice contains an unknown supplier item, show it as unmatched.
Admin must be able to match it to an INSHOP item.
Save the mapping in SupplierItemMapping.
Next time the same supplier item code/name appears, auto-match it.
Include audit log for mapping decisions.

5. Credit flow:
If shortage exists, system creates a credit request.
Supplier/accountant portal shows credit requests.
Supplier uploads credit invoice.
System parses credit invoice through mock AI parser.
System validates:
- supplier
- branch
- original invoice number if available
- credit amount
- credit line items
- whether it covers the requested credit
Admin can approve/reject credit invoice.
Approved credits become available for payment offset.

6. Payments flow:
Admin payments screen must show supplier invoices ready for payment.
Admin can select one or more debit invoices.
System calculates:
- gross invoice total
- approved credits to offset
- open credit requests
- net amount to pay
- previous balance
- remaining balance

Rule:
An invoice with an open credit request cannot be closed as fully paid unless admin explicitly approves an override with reason.

Admin uploads or records payment proof.
System parses/records payment details.
Status becomes paid, partially_paid, overpaid, or payment_pending_credit depending on scenario.
Show payment history and supplier balance.

7. Inventory count:
Implement the inventory count module from the spec.
Before count starts, call LegacyAdapter to simulate closing/checking open branch invoices.
Count can start only after adapter returns approval or allowed rejected invoices.
Count all items in units.
Each item must be saved separately with Swipe/confirm action.
Count cannot be completed until all items are saved.
Completed count is locked.
Admin can unlock only with audit.

8. Inventory formula:
Implement real mock inventory values.
Current inventory per branch/item must be calculated as:
last locked inventory count
+ admin-approved received quantities
- sales/delta movements from LegacyAdapter
+/- manual admin adjustments with audit

Do not show fake static inventory numbers.
Inventory must change after approved receiving, sales delta import and new count.

- package size
- supplier item code
- supplier item name
- WhatsApp numbers for orders
